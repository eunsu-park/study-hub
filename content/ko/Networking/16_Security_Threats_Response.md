# 보안 위협과 대응

**이전**: [네트워크 보안 기초](./15_Network_Security_Basics.md) | **다음**: [실용 네트워크 도구](./17_Practical_Network_Tools.md)

---

## 학습 목표(Learning Objectives)

이 레슨을 마치면 다음을 할 수 있습니다:

1. 네트워크 공격을 수동적(Passive) 또는 능동적(Active) 공격으로 분류하고, 각각이 표적으로 삼는 OSI 계층을 매핑한다
2. 스니핑(Sniffing)과 ARP 스푸핑(ARP Spoofing)의 동작 원리를 설명하고, 효과적인 대응 방안을 식별한다
3. IP 스푸핑(IP Spoofing), DNS 스푸핑(DNS Spoofing), 이메일 스푸핑(Email Spoofing) 기법을 구분한다
4. DoS/DDoS 공격(SYN Flood, UDP Flood, 증폭 공격)의 메커니즘과 다층적 방어 전략을 설명한다
5. SSL 스트리핑(SSL Stripping)과 Evil Twin을 포함한 중간자 공격(Man-in-the-Middle, MITM)을 설명하고, HSTS와 VPN이 이를 완화하는 방법을 설명한다
6. 웹 애플리케이션 위협(SQL Injection, XSS, CSRF)을 식별하고 방어적 코딩 관행을 적용한다
7. IDS와 IPS 배치 모델을 비교하고, 시그니처 기반(Signature-Based) 탐지와 이상 탐지(Anomaly-Based) 탐지를 구분한다
8. NIST 사고 대응 생명주기(준비부터 교훈 도출까지)를 개략적으로 설명한다

---

**난이도**: ⭐⭐⭐⭐

네트워크를 방어하는 방법을 이해하려면 공격자의 사고방식을 먼저 이해해야 합니다. 수동적인 도청에서 정교한 분산 서비스 거부(Distributed Denial-of-Service) 캠페인까지, 위협의 양상은 방대하고 끊임없이 진화합니다. 이 레슨에서는 가장 일반적인 공격 벡터(Attack Vector)를 순서대로 살펴보고, 각 메커니즘을 상세히 설명하며, 각각에 실질적인 대응 방안을 제시합니다 — 실제 보안 사고를 인식하고 대응하는 데 필요한 지식을 갖추도록 합니다.

## 2. 스니핑

### 스니핑 개요

스니핑(Sniffing)은 네트워크 트래픽을 가로채어 정보를 수집하는 수동적 공격입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    스니핑 공격                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  정상 통신:                                                     │
│                                                                 │
│  [클라이언트] ────────────────────────────────▶ [서버]          │
│               "사용자: admin, 비밀번호: 1234"                    │
│                                                                 │
│  스니핑 공격:                                                   │
│                                                                 │
│  [클라이언트] ────────────────────────────────▶ [서버]          │
│               "사용자: admin, 비밀번호: 1234"                    │
│                           │                                     │
│                           │ 도청                                │
│                           ▼                                     │
│                      [공격자]                                   │
│                  "자격 증명 획득!"                               │
│                                                                 │
│  탈취 가능한 정보:                                               │
│  - 사용자 계정 정보                                              │
│  - 이메일 내용                                                  │
│  - 금융 정보                                                    │
│  - 세션 토큰                                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 스니핑 유형

#### 1. 패시브 스니핑 (허브 환경)

```
┌─────────────────────────────────────────────────────────────────┐
│                    허브 환경 스니핑                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  허브(Hub)는 모든 포트에 패킷을 브로드캐스트                      │
│                                                                 │
│        ┌────────────────────────────────┐                      │
│        │             허브               │                      │
│        │  (모든 포트에 패킷 전송)        │                      │
│        └─┬──────┬──────┬──────┬────────┘                      │
│          │      │      │      │                                │
│          ▼      ▼      ▼      ▼                                │
│        ┌──┐   ┌──┐   ┌──┐   ┌──┐                              │
│        │PC│   │PC│   │PC│   │공격자│                           │
│        │A │   │B │   │C │   │ D  │                            │
│        └──┘   └──┘   └──┘   └──┘                              │
│                              ↑                                 │
│                         모든 트래픽                             │
│                         수신 가능                               │
│                                                                 │
│  * 현재 허브는 거의 사용되지 않음                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 2. 액티브 스니핑 (스위치 환경)

```
┌─────────────────────────────────────────────────────────────────┐
│                   스위치 환경 스니핑 기법                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  스위치는 MAC 주소 테이블로 목적지 포트로만 전송                   │
│  → 추가 기법이 필요                                              │
│                                                                 │
│  1. ARP 스푸핑/포이즈닝                                          │
│     - 가짜 ARP 응답으로 MAC 테이블 조작                          │
│     - 트래픽이 공격자를 경유하도록 유도                           │
│                                                                 │
│  2. MAC Flooding                                                │
│     - 대량의 가짜 MAC 주소로 스위치 테이블 넘침                   │
│     - 스위치가 허브처럼 동작                                     │
│                                                                 │
│  3. SPAN/Mirror Port                                            │
│     - 스위치의 모니터링 포트 악용 (내부자 위협)                   │
│                                                                 │
│  4. DHCP 스푸핑                                                 │
│     - 가짜 DHCP 서버로 게이트웨이 조작                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ARP 스푸핑 상세

```
┌─────────────────────────────────────────────────────────────────┐
│                    ARP 스푸핑 공격                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  정상 상태:                                                     │
│                                                                 │
│  [피해자 PC]              [게이트웨이]                           │
│  IP: 192.168.1.10        IP: 192.168.1.1                        │
│  MAC: AA:AA:AA           MAC: BB:BB:BB                          │
│       │                        │                                │
│       │◀─────── 정상 통신 ──────▶│                                │
│                                                                 │
│  ARP 스푸핑 후:                                                 │
│                                                                 │
│  [피해자 PC]        [공격자]         [게이트웨이]                 │
│  192.168.1.10      192.168.1.100    192.168.1.1                 │
│  AA:AA:AA          CC:CC:CC          BB:BB:BB                   │
│       │                │                 │                      │
│       │                │                 │                      │
│  ARP 테이블:       가짜 ARP 전송:     ARP 테이블:                 │
│  ┌───────────┐    "192.168.1.1의     ┌───────────┐             │
│  │192.168.1.1│     MAC은 CC:CC:CC"   │192.168.1.10│             │
│  │→ CC:CC:CC │    "192.168.1.10의    │→ CC:CC:CC │             │
│  └───────────┘     MAC은 CC:CC:CC"   └───────────┘             │
│       │                │                 │                      │
│       │                │                 │                      │
│       └───────▶ 공격자 ◀────────────────┘                       │
│                 (중계 및 도청)                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 스니핑 대응 방안

| 대응 방법 | 왜 효과적인가 |
|----------|------|
| 암호화 사용 | HTTPS, SSH, VPN은 페이로드(Payload)를 암호화하므로, 공격자가 패킷을 캡처하더라도 세션 키(Session Key) 없이는 데이터를 읽을 수 없습니다 |
| 동적 ARP 검사 (DAI) | 스위치가 ARP 패킷을 신뢰할 수 있는 DHCP 스누핑(Snooping) 바인딩 테이블과 대조 검증하여, 트래픽을 공격자로 리디렉션하는 가짜 ARP 응답을 차단합니다 |
| 정적 ARP 테이블 | 중요 서버의 IP-MAC 매핑을 하드코딩하면 OS가 충돌하는 ARP 브로드캐스트를 무시하므로 ARP 포이즈닝(Poisoning)에 면역이 됩니다 |
| 802.1X | 포트가 활성화되기 전에 인증을 요구하여, 미승인 장치가 스니핑을 위해 네트워크에 참여하는 것 자체를 방지합니다 |
| 네트워크 분리 | VLAN이 브로드캐스트 도메인(Broadcast Domain)을 제한하여, 하나의 VLAN에 있는 공격자가 다른 VLAN의 트래픽을 볼 수 없으므로 스니핑 성공 시 범위를 한정합니다 |
| IDS/IPS | 중복 ARP 응답이나 MAC 주소 플래핑(Flapping) 같은 ARP 스푸핑 진행 중 패턴을 탐지하여 신속한 대응을 가능하게 합니다 |

## 3. 스푸핑

### 스푸핑 개요

스푸핑(Spoofing)은 신원을 위조하여 신뢰를 악용하는 공격입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    스푸핑 유형                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │                       스푸핑 공격                          │ │
│  ├───────────────────────────────────────────────────────────┤ │
│  │                                                           │ │
│  │  IP 스푸핑                                                │ │
│  │  └─ 출발지 IP 주소 위조                                   │ │
│  │                                                           │ │
│  │  MAC 스푸핑                                               │ │
│  │  └─ 출발지 MAC 주소 위조                                  │ │
│  │                                                           │ │
│  │  ARP 스푸핑                                               │ │
│  │  └─ ARP 응답 위조로 IP-MAC 매핑 조작                      │ │
│  │                                                           │ │
│  │  DNS 스푸핑                                               │ │
│  │  └─ DNS 응답 위조로 악성 서버로 유도                       │ │
│  │                                                           │ │
│  │  이메일 스푸핑                                             │ │
│  │  └─ 발신자 주소 위조                                       │ │
│  │                                                           │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### IP 스푸핑

```
┌─────────────────────────────────────────────────────────────────┐
│                    IP 스푸핑                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [공격자]                                      [서버]            │
│  실제 IP: 10.0.0.100                          192.168.1.1       │
│       │                                           │             │
│       │  위조된 패킷                              │             │
│       │  ┌─────────────────────────┐             │             │
│       │  │ 출발지: 192.168.1.50    │             │             │
│       │  │ (위조, 신뢰 IP)         │             │             │
│       │  │ 목적지: 192.168.1.1     │             │             │
│       │  └─────────────────────────┘             │             │
│       │                                           │             │
│       └───────────────────────────────────────────▶             │
│                                                   │             │
│                    응답은 위조된 IP로              │             │
│  [피해자]◀────────────────────────────────────────┘             │
│  192.168.1.50                                                   │
│                                                                 │
│  용도:                                                          │
│  - DoS 공격 (반사 공격)                                          │
│  - 접근 제어 우회                                                │
│  - 로깅 회피                                                    │
│                                                                 │
│  한계:                                                          │
│  - TCP 연결 수립 어려움 (3-way handshake)                       │
│  - 응답 수신 불가                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```

├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  정상 DNS 조회:                                                 │
│                                                                 │

│     │───────── 정상 접속 ───────────────────▶│                   │
│                                                                 │
│  DNS 스푸핑 공격:                                               │
│                                                                 │
│  [사용자]     [공격자]     [DNS 서버]    [bank.com]  [악성 서버] │

│     │     (더 빠르게 응답)                               │        │
│     │                                                   │        │
│     │───────────────── 악성 서버 접속 ─────────────────▶│        │

│                                                                 │
│  공격 방법:                                                     │
│  1. ARP 스푸핑 후 DNS 응답 조작                                  │
│  2. DNS 캐시 포이즈닝                                           │
│  3. 로컬 hosts 파일 변조                                        │
│  4. Rogue DNS 서버 (DHCP 스푸핑과 결합)                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 스푸핑 대응 방안

| 스푸핑 유형 | 대응 방법 | 왜 효과적인가 |
|------------|----------|----------|
| IP 스푸핑 | 인그레스/이그레스 필터링(Ingress/Egress Filtering), BCP38 | 라우터가 출발지 네트워크에 속하지 않는 출발지 IP의 패킷을 차단하여, 소스에서 주소 위조 능력을 제거합니다 |
| MAC 스푸핑 | 802.1X, 포트 보안(Port Security) | 포트 보안은 각 스위치 포트를 특정 MAC 주소로 제한하고, 802.1X는 접근 전 장치를 인증하여 유효한 자격 증명(Credentials) 없이는 사칭이 불가능합니다 |
| ARP 스푸핑 | DAI, 정적 ARP, ARP 워치(ARP Watch) | DAI는 ARP 패킷을 DHCP 바인딩과 교차 참조하고, 정적 항목은 브로드캐스트로 덮어쓸 수 없으며, ARP 워치는 중요 매핑 변경 시 알림을 보냅니다 |
| DNS 스푸핑 | DNSSEC, DoH/DoT, DNS 모니터링 | DNSSEC는 DNS 레코드에 암호화 서명(Cryptographic Signature)을 하여 위조된 응답이 검증에 실패하고, DoH/DoT는 쿼리를 암호화하여 가로채기와 주입을 방지합니다 |
| 이메일 스푸핑 | SPF, DKIM, DMARC | SPF는 도메인에 대해 발송 가능한 서버를 지정하고, DKIM은 메시지 헤더에 암호화 서명을 하며, DMARC는 SPF/DKIM 실패 시 수신자에게 조치를 지시합니다 -- 함께 사용하면 발신자 주소 위조를 검증 가능하게 탐지합니다 |

## 4. DoS/DDoS 공격

### DoS 개요

DoS(Denial of Service)는 시스템이나 네트워크 자원을 고갈시켜 정상 서비스를 방해하는 공격입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    DoS vs DDoS                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  DoS (단일 출처)                    DDoS (분산 출처)             │
│                                                                 │
│     [공격자]                          [공격자]                   │
│        │                                 │                      │
│        │                          ┌──────┼──────┐               │
│        │                          │      │      │               │
│        ▼                         ▼      ▼      ▼              │
│     [대상]                     [봇] [봇] [봇] [봇]             │
│                                   │      │      │               │
│                                   └──────┼──────┘               │
│                                          │                      │
│                                          ▼                      │
│                                       [대상]                    │
│                                                                 │
│  특징:                              특징:                       │
│  - 단일 IP 차단으로 방어 가능        - 다수의 출처로 차단 어려움   │
│  - 대역폭 한계                       - 봇넷 활용                 │
│                                     - 대규모 트래픽 생성         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### DoS 공격 유형

#### 1. TCP SYN Flood

```
┌─────────────────────────────────────────────────────────────────┐
│                    TCP SYN Flood                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  정상 TCP 3-way Handshake:                                      │
│                                                                 │
│  [클라이언트]                              [서버]                │
│      │                                        │                 │
│      │──── SYN ────────────────────────────▶ │                 │
│      │                                        │ 연결 대기       │
│      │◀─── SYN-ACK ──────────────────────── │ (리소스 할당)    │
│      │                                        │                 │
│      │──── ACK ────────────────────────────▶ │                 │
│      │                                        │ 연결 완료       │
│                                                                 │
│  SYN Flood 공격:                                                │
│                                                                 │
│  [공격자]                                  [서버]               │
│      │                                        │                 │
│      │──── SYN (위조 IP) ──────────────────▶ │                 │
│      │──── SYN (위조 IP) ──────────────────▶ │ Half-open       │
│      │──── SYN (위조 IP) ──────────────────▶ │ 연결 누적       │
│      │──── SYN (위조 IP) ──────────────────▶ │                 │
│      :           ×1000                        │ 리소스 고갈     │
│                                              │                 │
│                                        ┌─────┴─────┐           │
│                                        │ 연결 테이블 │           │
│                                        │ 가득 참    │           │
│                                        │           │           │
│                                        │ 정상 연결  │           │
│                                        │ 불가능    │           │
│                                        └───────────┘           │
│                                                                 │
│  대응:                                                           │
│  - SYN Cookies: 각 SYN에 대해 메모리를 할당하는 대신, 서버가      │
│    연결 상태(Connection State)를 SYN-ACK 시퀀스 번호 자체에       │
│    인코딩합니다. 클라이언트가 유효한 ACK로 핸드셰이크를 완료할    │
│    때까지 메모리가 소비되지 않아, 연결 테이블(Connection Table)    │
│    고갈이라는 공격의 핵심 전략을 무력화합니다.                     │
│  - 연결 제한: 출발지 IP당 반개방(Half-open) 연결 수를 제한합니다. │
│    정상 클라이언트는 동시에 소수의 연결만 여는 반면, 공격자는      │
│    수천 개를 플러딩하기 때문입니다.                                │
│  - 방화벽 필터링: 인그레스 필터링(Ingress Filtering, BCP38)으로   │
│    스푸핑된 IP의 SYN 패킷을 차단하여, 공격자가 실제 주소에 연결된 │
│    속도 제한(Rate Limit)을 회피하는 데 사용하는 익명성을 제거합니다.│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 2. UDP Flood

```
┌─────────────────────────────────────────────────────────────────┐
│                    UDP Flood                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [공격자/봇넷]                                                  │
│       │                                                         │
│       │  대량의 UDP 패킷                                        │
│       │  (무작위 포트)                                          │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                        대상 서버                          │   │
│  │                                                         │   │
│  │   UDP 포트 확인 → 서비스 없음 → ICMP 응답 생성           │   │
│  │                                                         │   │
│  │   반복 처리로 CPU/대역폭 소진                             │   │
│  │                                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  특징:                                                          │
│  - 비연결형 프로토콜 악용                                        │
│  - 대역폭 포화                                                  │
│  - 출발지 IP 스푸핑 용이                                         │
│                                                                 │
│  대응:                                                           │
│  - Rate Limiting: 수신 대기하지 않는 UDP 포트로의 각 패킷은 서버  │
│    에게 ICMP "port unreachable" 응답을 생성하도록 강제하여 CPU를  │
│    소비합니다. ICMP 응답을 속도 제한하면 이 CPU 소모를 억제합니다. │
│  - 블랙홀 라우팅(Blackhole Routing): 플러드 트래픽을 널 라우트    │
│    (Drop)로 리디렉션합니다. 대상 IP의 도달 가능성을 희생하지만    │
│    나머지 네트워크를 대역폭 포화(Bandwidth Saturation)로부터      │
│    보호합니다.                                                    │
│  - UDP 서비스 최소화: 열린 모든 UDP 포트는 공격 표면(Attack       │
│    Surface)입니다. 불필요한 포트를 닫으면 서버가 해당 패킷을      │
│    처리하고 응답해야 할 의무 자체가 완전히 사라집니다.             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 3. Amplification Attack (증폭 공격)

```
┌─────────────────────────────────────────────────────────────────┐
│                    증폭 공격 (DNS, NTP)                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  DNS Amplification:                                             │
│                                                                 │
│  [공격자]                                                       │
│      │                                                          │
│      │  작은 요청 (60 bytes)                                    │
│      │  출발지: 피해자 IP (스푸핑)                               │
│      ▼                                                          │
│  ┌─────────┐                                                   │
│  │ 오픈    │  큰 응답 (3000 bytes)                              │
│  │ DNS    │──────────────────────────────▶ [피해자]            │
│  │ 서버    │  증폭 비율: 50배                                   │
│  └─────────┘                                                   │
│                                                                 │
│  ┌────────────────────────────────────────────────────────────┐│
│  │  프로토콜    │  증폭 비율  │  사용 포트                      ││
│  ├────────────────────────────────────────────────────────────┤│
│  │  DNS        │  28-54x    │  UDP 53                        ││
│  │  NTP        │  556x      │  UDP 123                       ││
│  │  SSDP       │  30x       │  UDP 1900                      ││
│  │  Memcached  │  51,000x   │  UDP 11211                     ││
│  └────────────────────────────────────────────────────────────┘│
│                                                                 │
│  대응:                                                          │
│  - 오픈 리졸버(Open Resolver) 차단: 증폭 공격은 어떤 출발지 IP  │
│    에도 응답하는 공개 DNS/NTP 서버에 의존합니다. 쿼리를 허가된   │
│    클라이언트로 제한하면 60바이트 요청을 3,000바이트 플러드로    │
│    변환하는 증폭기(Amplifier)를 제거합니다.                     │
│  - BCP38 (인그레스 필터링): ISP가 네트워크 경계에서 스푸핑된    │
│    출발지 IP 패킷을 차단합니다. 증폭 공격은 피해자의 IP를       │
│    출발지로 위조하는 것에 의존하므로, 소스 네트워크에서의        │
│    필터링은 공격이 시작되기 전에 차단합니다.                     │
│  - Response Rate Limiting (RRL): DNS 서버가 동일 목적지에 대한  │
│    동일 응답을 스로틀(Throttle)하여, 증폭기가 악용되더라도      │
│    플러드 볼륨이 관리 가능한 수준으로 제한됩니다.               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### DDoS 공격 대응

> **왜 계층별 방어(Layered Defense)인가?** 단일 대응 방법으로는 모든 DDoS 공격 유형을 처리할 수 없습니다. 볼류메트릭 공격(Volumetric Attack, UDP 플러드)은 대역폭을 압도합니다 -- 대규모 용량을 가진 업스트림 제공자(Upstream Provider)만이 이를 흡수할 수 있습니다. 프로토콜 공격(Protocol Attack, SYN 플러드)은 서버 상태 테이블을 고갈시킵니다 -- 연결 수준의 인식을 가진 로컬 장비만이 SYN 쿠키를 적용할 수 있습니다. 애플리케이션 계층 공격(Application Layer Attack, HTTP 플러드)은 정상 요청을 모방합니다 -- 애플리케이션을 인식하는 도구(WAF, CAPTCHA)만이 이를 구분할 수 있습니다. 각 계층이 다른 계층이 처리할 수 없는 것을 담당하여, 심층 방어(Defense in Depth)를 구현합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    DDoS 방어 전략                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  계층별 방어:                                                   │
│                                                                 │
│  [인터넷]                                                       │
│     │                                                           │
│     ▼                                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. CDN/클라우드 기반 보호                                │   │
│  │    - Cloudflare, AWS Shield, Akamai                     │   │
│  │    - 분산 처리, 대역폭 흡수                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│     │                                                           │
│     ▼                                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 2. ISP 레벨 필터링                                       │   │
│  │    - 블랙홀 라우팅                                       │   │
│  │    - 업스트림 필터링                                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│     │                                                           │
│     ▼                                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 3. 온프레미스 장비                                       │   │
│  │    - DDoS 방어 장비                                      │   │
│  │    - 방화벽/IPS                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
│     │                                                           │
│     ▼                                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 4. 애플리케이션 레벨                                      │   │
│  │    - Rate Limiting                                       │   │
│  │    - CAPTCHA                                            │   │
│  │    - WAF                                                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 5. MITM 공격

### MITM 개요

MITM(Man-in-the-Middle)은 두 당사자 간의 통신을 가로채어 도청하거나 변조하는 공격입니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    MITM 공격                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  정상 통신:                                                     │
│                                                                 │
│  [클라이언트] ◀═══════════════════════════════▶ [서버]          │
│                      직접 통신                                   │
│                                                                 │
│  MITM 공격:                                                     │
│                                                                 │
│  [클라이언트]        [공격자]         [서버]                     │
│       │                │                │                       │
│       │════════════════│════════════════│                       │
│       │    가짜 연결    │    가짜 연결    │                       │
│       │                │                │                       │
│       │──"계좌이체"───▶│                │                       │
│       │                │──"계좌이체"───▶│                       │
│       │                │(내용 변조 가능) │                       │
│       │                │                │                       │
│       │                │◀──"완료"──────│                       │
│       │◀──"완료"──────│                │                       │
│                                                                 │
│  공격자 능력:                                                   │
│  - 모든 통신 내용 도청                                          │
│  - 데이터 변조                                                  │
│  - 세션 하이재킹                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

| 대응 방법 | 왜 효과적인가 |
|----------|------|
| TLS/SSL 사용 | 종단 간 암호화(End-to-End Encryption)로 공격자가 패킷을 가로채더라도 핸드셰이크 중 협상된 세션 키 없이는 내용을 복호화할 수 없습니다 |
| 인증서 검증 | 서버 인증서를 신뢰할 수 있는 CA에 대해 검증하여 사칭 서버를 탐지합니다; MITM은 CA를 침해하지 않고는 유효한 인증서를 위조할 수 없습니다 |
| HSTS | 브라우저가 사이트에 HTTPS가 필요하다는 것을 기억하고 HTTP 연결을 거부하여, 암호화 시작 전에 연결을 다운그레이드하는 SSL 스트리핑(SSL Stripping) 공격을 방지합니다 |
| Certificate Pinning | 클라이언트가 미리 구성된 인증서나 공개키만 수락하므로, 공격자가 위조 CA 서명 인증서를 획득하더라도 거부됩니다 |
| VPN | 장치에서 VPN 서버까지 모든 트래픽을 암호화하여, MITM이 주로 발생하는 로컬 네트워크(Local Network)가 무관해집니다 -- 공격자는 불투명한 암호문(Ciphertext)만 볼 수 있습니다 |
| 2FA | 공격자가 MITM을 통해 비밀번호를 캡처하더라도 두 번째 인증 요소(TOTP 토큰, 하드웨어 키) 없이는 인증할 수 없어, 가로챈 자격 증명의 피해를 제한합니다 |

### 안전한 코드 예시

```python
# 취약한 코드 (Python)

# 안전한 코드 (Prepared Statement)

```

```java
// 취약한 코드 (Java)
String query = "SELECT * FROM users WHERE id = '" + userId + "'";
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(query);

// 안전한 코드 (PreparedStatement)
String query = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = conn.prepareStatement(query);
pstmt.setString(1, userId);
ResultSet rs = pstmt.executeQuery();
```

### IDS 유형

| 유형 | 설명 | 장점 | 단점 |
|------|------|------|------|
| NIDS | 네트워크 트래픽 분석 | 전체 트래픽 모니터링 | 암호화 트래픽 분석 어려움 |
| HIDS | 호스트 활동 분석 | 상세한 분석 가능 | 호스트별 설치 필요 |
| 시그니처 기반 | 알려진 패턴 매칭 | 알려진 공격 정확 탐지 | 제로데이 탐지 불가 |
| 이상 탐지 | 정상 패턴 벗어나면 탐지 | 새로운 공격 탐지 가능 | 오탐(False Positive) 높음 |
