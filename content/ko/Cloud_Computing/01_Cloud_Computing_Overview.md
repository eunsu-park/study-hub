클라우드 컴퓨팅은 조직이 기술을 구축하고 운영하는 방식을 근본적으로 바꾸었습니다. 서버를 구매하고 데이터 센터를 관리하는 대신, 팀은 몇 분 만에 인프라를 구축하고 사용한 만큼만 비용을 지불할 수 있습니다. 현대 소프트웨어 시스템을 다루는 모든 엔지니어, 아키텍트, 의사결정자에게 핵심 개념, 서비스 모델, 배포 모델에 대한 이해는 필수입니다.

> **비유 — 발전기 임대 vs. 발전소 건설**: 자체 서버를 운영하는 것은 사설 발전소를 짓는 것과 같습니다. 막대한 초기 비용, 직접 유지·관리, 수요가 낮을 때도 비용이 발생합니다. 클라우드 컴퓨팅은 전력망에서 전기를 임대하는 것과 같습니다 — 플러그를 꽂고, kWh당 요금을 내고, 용량 관리는 제공업체가 담당합니다. IaaS는 날 전기(VM)를 제공하고, PaaS는 전압 조절기가 달린 콘센트(관리형 플랫폼)를 제공하며, SaaS는 사전 배선된 가전제품(완성된 소프트웨어)을 제공합니다.

## 1. 클라우드 컴퓨팅이란?

클라우드 컴퓨팅은 인터넷을 통해 IT 리소스(서버, 스토리지, 네트워크, 데이터베이스 등)를 온디맨드로 제공하는 서비스 모델입니다.

### 1.1 전통적 인프라 vs 클라우드

| 구분 | 온프레미스 (전통) | 클라우드 |
|------|------------------|----------|
| **초기 비용** | 높음 (하드웨어 구매) | 낮음 (사용량 기반) |
| **확장성** | 수주~수개월 | 수분~수시간 |
| **유지보수** | 직접 관리 | 제공자가 관리 |
| **위험** | 미사용 리소스 낭비 | 필요한 만큼만 사용 |
| **책임** | 모든 계층 직접 관리 | 공동 책임 모델 |

### 1.2 NIST 클라우드의 5가지 특성

미국 국립표준기술연구소(NIST)가 정의한 클라우드 컴퓨팅의 핵심 특성:

1. **On-demand Self-service (온디맨드 셀프서비스)**
   - 사용자가 직접 리소스를 프로비저닝
   - 사람의 개입 없이 자동화된 배포

2. **Broad Network Access (광범위한 네트워크 접근)**
   - 표준 메커니즘으로 네트워크를 통해 접근
   - 다양한 클라이언트 플랫폼 지원

3. **Resource Pooling (리소스 풀링)**
   - 멀티테넌트 모델로 리소스 공유
   - 물리적 위치 추상화

4. **Rapid Elasticity (빠른 탄력성)**
   - 수요에 따라 자동 확장/축소
   - 사용자에게는 무한한 리소스처럼 보임

5. **Measured Service (측정된 서비스)**
   - 사용량 모니터링 및 보고
   - 투명한 과금

## 2. 서비스 모델: IaaS, PaaS, SaaS

### 2.1 개념 비교

```
┌─────────────────────────────────────────────────────────────┐
│                        SaaS                                 │
│  (Software as a Service)                                    │
│  예: Gmail, Salesforce, Slack                               │
├─────────────────────────────────────────────────────────────┤
│                        PaaS                                 │
│  (Platform as a Service)                                    │
│  예: Heroku, App Engine, Elastic Beanstalk                  │
├─────────────────────────────────────────────────────────────┤
│                        IaaS                                 │
│  (Infrastructure as a Service)                              │
│  예: EC2, Compute Engine, Azure VMs                         │
├─────────────────────────────────────────────────────────────┤
│                   Physical Infrastructure                    │
│  데이터센터, 서버, 네트워크 장비                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 책임 범위 비교

| 계층 | 온프레미스 | IaaS | PaaS | SaaS |
|------|-----------|------|------|------|
| 애플리케이션 | 고객 | 고객 | 고객 | **제공자** |
| 데이터 | 고객 | 고객 | 고객 | 고객* |
| 런타임 | 고객 | 고객 | **제공자** | **제공자** |
| 미들웨어 | 고객 | 고객 | **제공자** | **제공자** |
| OS | 고객 | 고객 | **제공자** | **제공자** |
| 가상화 | 고객 | **제공자** | **제공자** | **제공자** |
| 서버 | 고객 | **제공자** | **제공자** | **제공자** |
| 스토리지 | 고객 | **제공자** | **제공자** | **제공자** |
| 네트워킹 | 고객 | **제공자** | **제공자** | **제공자** |

*SaaS에서도 데이터 관리 책임은 고객에게 있음

### 2.3 사용 사례

**IaaS 적합 사례:**
- 완전한 인프라 제어가 필요한 경우
- 레거시 애플리케이션 마이그레이션
- 개발/테스트 환경
- 고성능 컴퓨팅 (HPC)

**PaaS 적합 사례:**
- 빠른 애플리케이션 개발
- 마이크로서비스 아키텍처
- API 개발
- 인프라 관리 부담 최소화

**SaaS 적합 사례:**
- 이메일, 협업 도구
- CRM, ERP 시스템
- 즉시 사용 가능한 솔루션 필요

## 4. 과금 모델

### 4.1 과금 원칙

두 플랫폼 모두 **사용한 만큼 지불 (Pay-as-you-go)** 원칙을 따릅니다.

```
총 비용 = 컴퓨팅 + 스토리지 + 네트워크 + 추가 서비스
```

### 4.2 컴퓨팅 과금 옵션

| 옵션 | AWS | GCP | 특징 |
|------|-----|-----|------|
| **온디맨드** | On-Demand | On-demand | 시간/초 단위, 약정 없음 |
| **예약** | Reserved Instances | Committed Use | 1-3년 약정, 최대 72% 할인 |
| **스팟/선점형** | Spot Instances | Preemptible/Spot VMs | 최대 90% 할인, 중단 가능 |
| **자동 할인** | - | Sustained Use | 월 사용량에 따라 자동 할인 |

### 4.3 데이터 전송 비용

```
┌─────────────────────────────────────────────────────────┐
│                        클라우드                         │
│                                                         │
│   ┌─────────┐         ┌─────────┐         ┌─────────┐  │
│   │ 인바운드 │   무료   │  같은   │   무료   │아웃바운드│  │
│   │(외부→내부)│ ────→  │  리전   │ ────→  │(내부→외부)│  │
│   └─────────┘         └─────────┘         └─────────┘  │
│       무료              무료/저렴            유료        │
└─────────────────────────────────────────────────────────┘
```

- **인바운드**: 일반적으로 무료
- **같은 리전 내**: 무료 또는 저렴
- **아웃바운드**: GB당 과금 (월 무료 할당량 후)

### 4.4 무료 티어

**AWS 무료 티어:**
- 12개월 무료: t2.micro EC2 (750시간/월), 5GB S3, 750시간 RDS
- 항상 무료: Lambda 100만 요청/월, DynamoDB 25GB

**GCP 무료 티어:**
- 90일 $300 크레딧 (신규 계정)
- Always Free: e2-micro VM, 5GB Cloud Storage, Cloud Functions 200만 호출/월

## 5. 공동 책임 모델 (Shared Responsibility Model)

클라우드 보안은 제공자와 고객이 역할을 분담합니다.

### 5.1 책임 분담

```
┌────────────────────────────────────────────────────────────┐
│                     고객 책임 (IN the cloud)               │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  • 고객 데이터                                        │  │
│  │  • 플랫폼, 애플리케이션, IAM                          │  │
│  │  • 운영체제, 네트워크, 방화벽 구성                    │  │
│  │  • 클라이언트 측 데이터 암호화                        │  │
│  │  • 서버 측 암호화 (파일 시스템/데이터)                │  │
│  │  • 네트워크 트래픽 보호 (암호화, 무결성, 인증)        │  │
│  └──────────────────────────────────────────────────────┘  │
├────────────────────────────────────────────────────────────┤
│                    제공자 책임 (OF the cloud)              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  • 글로벌 인프라 (리전, 가용 영역, 엣지 로케이션)     │  │
│  │  • 하드웨어 (컴퓨팅, 스토리지, 네트워킹)              │  │
│  │  • 소프트웨어 (호스트 운영체제, 가상화)               │  │
│  │  • 물리적 보안 (데이터센터)                          │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

### 5.2 서비스 유형별 책임

| 서비스 유형 | 고객 책임 | 제공자 책임 |
|------------|----------|------------|
| **IaaS (EC2)** | OS부터 애플리케이션까지 | 하드웨어, 가상화 |
| **PaaS (Lambda)** | 코드, 데이터 | 런타임, OS, 인프라 |
| **SaaS** | 데이터, 접근 관리 | 거의 모든 것 |

## 6. 클라우드 아키텍처 원칙

### 6.1 Well-Architected Framework

AWS와 GCP 모두 비슷한 설계 원칙을 제시합니다:

| 원칙 | 설명 |
|------|------|
| **운영 우수성** | 시스템 실행 및 모니터링, 지속적 개선 |
| **보안** | 데이터, 시스템, 자산 보호 |
| **신뢰성** | 장애 복구, 수요 변화 대응 |
| **성능 효율성** | 리소스 효율적 사용, 기술 선택 |
| **비용 최적화** | 불필요한 비용 제거, 효율적 지출 |
| **지속 가능성** | 환경 영향 최소화 (GCP 강조) |

### 6.2 AWS Well-Architected Framework: 6가지 기둥(Pillar)

AWS Well-Architected Framework는 클라우드 모범 사례에 따라 아키텍처를 평가하는 체계적인 접근법을 제공합니다. 6개의 기둥으로 구성되며, 각각 핵심 설계 원칙이 있습니다:

| 기둥(Pillar) | 설명 | 핵심 설계 원칙 |
|--------|-------------|----------------------|
| **운영 우수성(Operational Excellence)** | 비즈니스 가치를 제공하기 위해 시스템을 실행·모니터링하고, 프로세스와 절차를 지속적으로 개선 | 1. 운영을 코드로 수행 (IaC, 런북) 2. 작고 되돌릴 수 있는 변경을 자주 수행 |
| **보안(Security)** | 위험 평가와 완화 전략을 통해 데이터, 시스템, 자산을 보호 | 1. 모든 계층에 보안 적용 (심층 방어) 2. 보안 모범 사례를 자동화 |
| **신뢰성(Reliability)** | 워크로드가 기대한 대로 정확하고 일관되게 의도된 기능을 수행하도록 보장 | 1. 장애로부터 자동 복구 2. 수평 확장으로 가용성 극대화 |
| **성능 효율성(Performance Efficiency)** | 시스템 요구사항을 충족하고 수요 변화에 따른 효율성을 유지하기 위해 컴퓨팅 리소스를 효율적으로 사용 | 1. 고급 기술의 민주화 (관리형 서비스 활용) 2. 몇 분 만에 글로벌 배포 |
| **비용 최적화(Cost Optimization)** | 불필요한 비용을 피하고 요구사항을 충족하면서 최저 가격으로 시스템을 운영 | 1. 클라우드 재무 관리 도입 2. 소비 모델 채택 (사용한 만큼만 지불) |
| **지속 가능성(Sustainability)** | 에너지 효율적인 방식으로 클라우드 워크로드 운영의 환경적 영향을 최소화 | 1. 영향 파악 (측정 및 추적) 2. 활용률 극대화 (리소스 적정 크기 조정) |

#### Well-Architected 리뷰(Review) 프로세스

Well-Architected Review(WAR)는 프레임워크에 따라 워크로드를 체계적으로 평가하는 과정입니다:

```
┌────────────────────────────────────────────────────────────┐
│              Well-Architected 리뷰 프로세스                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 범위 설정 → 리뷰할 워크로드 식별                        │
│  2. 리뷰     → 각 기둥별 질문에 답변                        │
│                (AWS WAR 도구가 안내 질문을 제공)             │
│  3. 식별     → 고위험 이슈(HRI)와 중위험 이슈(MRI) 발견    │
│  4. 우선순위 → 비즈니스 영향도 기준으로 개선사항 순위 매기기 │
│  5. 개선     → 실행 계획 수립 및 수정 구현                   │
│  6. 측정     → 시간 경과에 따른 개선사항 추적                │
│  7. 반복     → 주기적으로 재리뷰 (분기별 또는 주요 변경 후) │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

> **팁**: AWS는 콘솔에서 **AWS Well-Architected Tool**을 제공합니다. 리뷰 질문을 대화형으로 안내하며 우선순위가 매겨진 개선 계획을 생성합니다.

#### 클라우드 제공업체 간 아키텍처 프레임워크 비교

주요 클라우드 제공업체마다 자체 아키텍처 프레임워크를 제공합니다. 핵심 원칙은 겹치지만 강조점이 다릅니다:

| 항목 | AWS Well-Architected | GCP Architecture Framework | Azure Well-Architected |
|--------|---------------------|---------------------------|----------------------|
| **기둥/영역** | 6개 기둥 | 6개 핵심 영역 (동일 명칭) | 5개 기둥 (지속 가능성 별도 분리 없음) |
| **리뷰 도구** | AWS WAR Tool (콘솔) | Architecture Framework 리뷰 (문서 기반) | Azure Advisor + WAF 평가 |
| **고유한 강조점** | 광범위한 렌즈 카탈로그 (SaaS, 서버리스, ML 등) | 데이터 중심 설계, BigQuery 우선 분석 | 하이브리드 클라우드 (Azure Arc), 엔터프라이즈 통합 |
| **렌즈/모듈** | 10개 이상의 전문 렌즈 | 산업별 블루프린트 | Azure 랜딩 존 |
| **자동화** | WAR Tool API, 커스텀 렌즈 | Terraform 블루프린트, Config Connector | Azure Policy, Blueprints |
| **커뮤니티** | AWS Partner 렌즈 프로그램 | Google Cloud Architecture Center | Azure Architecture Center |

### 6.3 재해 복구(Disaster Recovery) 전략

재해 복구(DR) 계획은 장애 발생 시 비즈니스 연속성을 보장합니다. 두 가지 핵심 지표는 다음과 같습니다:

- **RPO (Recovery Point Objective, 복구 시점 목표)**: 시간으로 측정한 최대 허용 데이터 손실량. "얼마나 많은 데이터를 잃을 수 있는가?"
- **RTO (Recovery Time Objective, 복구 시간 목표)**: 최대 허용 다운타임. "얼마나 빨리 복구해야 하는가?"

```
         비용 & 복잡도 →
  낮음 ─────────────────────────── 높음

  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Backup & │  │  Pilot   │  │  Warm    │  │Multi-Site│
  │ Restore  │  │  Light   │  │ Standby  │  │ Active-  │
  │          │  │          │  │          │  │ Active   │
  └──────────┘  └──────────┘  └──────────┘  └──────────┘
  RTO: 시간    RTO: 수십 분   RTO: 수 분    RTO: ~0
  RPO: 시간    RPO: 수 분     RPO: 수 초    RPO: ~0

  ← 복구 속도
  느림 ─────────────────────────── 빠름
```

#### DR 전략 비교

| 전략 | 설명 | RPO | RTO | 상대 비용 | 적합한 경우 |
|----------|-------------|-----|-----|---------------|----------|
| **Backup & Restore** | 다른 리전에 정기 백업; 재해 시 복원 | 시간 단위 | 시간 단위 | $ (최저) | 비핵심 워크로드, 개발/테스트 |
| **Pilot Light** | 최소한의 핵심 인프라 항시 가동 (DB 복제); 재해 시 확장 | 분 단위 | 수십 분 | $$ | 빠른 복구가 필요한 핵심 시스템 |
| **Warm Standby** | 축소되었지만 완전히 기능하는 프로덕션 복제본; 재해 시 확장 | 초~분 단위 | 분 단위 | $$$ | 비즈니스 크리티컬 애플리케이션 |
| **Multi-Site Active-Active** | 2개 이상 리전에서 풀 프로덕션; 모든 리전에 트래픽 분산 | 거의 0 | 거의 0 | $$$$ (최고) | 미션 크리티컬, 제로 다운타임 필수 |

#### 각 DR 전략별 AWS 서비스

| 전략 | 주요 AWS 서비스 |
|----------|-----------------|
| **Backup & Restore** | S3 (교차 리전 복제), AWS Backup, EBS 스냅샷, RDS 자동 백업, Glacier (아카이브용) |
| **Pilot Light** | Route 53 (DNS 장애 조치), RDS 읽기 복제본 (교차 리전), AMI를 DR 리전으로 복사, CloudFormation (빠른 스케일업) |
| **Warm Standby** | Auto Scaling (DR에서 최소 용량), Elastic Load Balancing, RDS Multi-AZ + 교차 리전 읽기 복제본, Route 53 헬스 체크 |
| **Multi-Site Active-Active** | Route 53 지연 시간 기반 라우팅, Global Accelerator, DynamoDB Global Tables, Aurora Global Database, S3 교차 리전 복제 |

> **비유 -- DR 전략을 집 안전에 비유하기**: **Backup & Restore**는 주택 보험에 가입하는 것과 같습니다 -- 재해 후 재건하지만 시간이 걸립니다. **Pilot Light**는 차고에 발전기를 준비해두어 바로 가동할 수 있게 하는 것입니다. **Warm Standby**는 부분적으로 가구가 배치되고 난방이 되는 세컨드 하우스를 유지하는 것입니다. **Multi-Site Active-Active**는 완전히 가구가 갖춰진 두 채의 집에서 동시에 생활하면서 더 편리한 곳을 사용하는 것입니다.

#### 올바른 전략 선택

적합한 DR 전략은 비즈니스 요구사항에 따라 달라집니다:

```
┌──────────────────────────────────────────────────────────┐
│           DR 전략 결정 요소                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 비즈니스 영향 분석                                    │
│     └─ 시간당 다운타임으로 인한 매출/평판 손실은?         │
│                                                          │
│  2. 데이터 중요도                                         │
│     └─ 데이터 손실을 감당할 수 있는가? 얼마나?            │
│                                                          │
│  3. 예산 제약                                             │
│     └─ DR 비용은 비즈니스 위험에 비례해야 함              │
│                                                          │
│  4. 규제 요구사항                                         │
│     └─ 일부 산업은 특정 RPO/RTO를 의무화                  │
│        (예: 금융 서비스, 헬스케어)                         │
│                                                          │
│  5. 기술적 복잡도                                         │
│     └─ DR 인프라를 유지할 팀 역량                         │
│                                                          │
│  경험 법칙:                                               │
│  Backup & Restore에서 시작하고, 비즈니스 요구사항이       │
│  요구할 때만 상위 전략으로 이동하세요.                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 6.4 설계 모범 사례

```
1. 장애에 대비한 설계 (Design for Failure)
   - 단일 장애점 제거
   - 멀티 AZ/리전 배포
   - 자동 복구

2. 컴포넌트 분리 (Loose Coupling)
   - 마이크로서비스 아키텍처
   - 메시지 큐 활용
   - API 기반 통신

3. 탄력성 (Elasticity)
   - 오토스케일링 활용
   - 서버리스 고려
   - 예측 불가능한 부하 대비

4. 보안 내재화 (Security by Design)
   - 최소 권한 원칙
   - 암호화 기본 적용
   - 네트워크 격리
```

## 연습 문제

### 연습 문제 1: 서비스 모델(Service Model) 분류

다음 각 시나리오에서 IaaS, PaaS, SaaS 중 어느 것에 해당하는지 판단하고 이유를 설명하세요:

1. 한 스타트업이 `git push`로 코드를 Heroku에 배포하여 Node.js 웹 앱을 운영합니다.
2. 한 회사가 EC2 인스턴스를 임대하여 자체 데이터베이스 소프트웨어를 설치합니다.
3. 영업팀이 Salesforce CRM을 사용하여 고객 파이프라인을 관리합니다.
4. 데이터 엔지니어링 팀이 Google App Engine에서 Python ETL 서비스를 운영합니다.

<details>
<summary>정답 보기</summary>

1. **PaaS** — Heroku가 런타임(runtime), OS, 인프라를 모두 관리합니다. 개발자는 애플리케이션 코드만 제공합니다.
2. **IaaS** — EC2는 순수한 가상 머신을 제공합니다. 회사가 OS, 설치된 소프트웨어, 설정에 대한 책임을 집니다.
3. **SaaS** — Salesforce는 인터넷을 통해 제공되는 완전 관리형 소프트웨어입니다. 사용자는 기반 인프라를 전혀 관리하지 않습니다.
4. **PaaS** — App Engine이 런타임 환경을 관리합니다. 팀은 코드를 배포하고, Google이 스케일링(scaling), OS 패치, 인프라를 처리합니다.

</details>

### 연습 문제 2: 공동 책임 모델(Shared Responsibility Model) 매핑

한 회사가 AWS EC2에 웹 애플리케이션을 호스팅합니다. 각 보안 작업을 올바른 책임 주체(AWS 또는 고객)에 매핑하세요:

| 작업 | 책임 주체 |
|------|-----------|
| 호스트 하이퍼바이저(hypervisor) 패치 | ? |
| EC2 인스턴스 OS 보안 업데이트 설치 | ? |
| 데이터베이스에 저장된 애플리케이션 데이터 암호화(encryption) | ? |
| 데이터 센터 물리적 접근 통제 보장 | ? |
| 보안 그룹(security group) 방화벽 규칙 설정 | ? |

<details>
<summary>정답 보기</summary>

| 작업 | 책임 주체 |
|------|-----------|
| 호스트 하이퍼바이저(hypervisor) 패치 | **AWS** — AWS가 가상화 레이어(virtualization layer)를 관리합니다 |
| EC2 인스턴스 OS 보안 업데이트 설치 | **고객** — IaaS에서 OS는 고객의 책임입니다 |
| 데이터베이스에 저장된 애플리케이션 데이터 암호화(encryption) | **고객** — 데이터 보호는 항상 고객의 책임입니다 |
| 데이터 센터 물리적 접근 통제 보장 | **AWS** — 데이터 센터 물리적 보안은 AWS의 책임입니다 |
| 보안 그룹(security group) 방화벽 규칙 설정 | **고객** — IaaS에서 네트워크/방화벽 설정은 고객이 관리합니다 |

**핵심 개념**: IaaS에서 AWS는 클라우드 *자체*(물리적 인프라와 가상화)에 대해 책임지고, 고객은 클라우드 *안에서*(OS 이상 모든 것)에 대해 책임집니다.

</details>

### 연습 문제 3: 재해 복구(Disaster Recovery) 전략 선택

한 핀테크(fintech) 회사가 실시간 결제 트랜잭션(transaction)을 처리합니다. 30분의 서비스 중단은 약 50만 달러의 손실을 초래하며, 규제 기관은 트랜잭션 데이터를 단 하나도 손실하지 않도록 요구합니다. DR 인프라 예산은 충분합니다.

1. 어떤 DR 전략을 선택해야 합니까? RPO와 RTO 요건을 사용하여 이유를 설명하세요.
2. 이 전략의 핵심이 될 AWS 서비스는 무엇입니까?

<details>
<summary>정답 보기</summary>

1. **멀티사이트 액티브-액티브(Multi-Site Active-Active)** 전략이 적합합니다.
   - **RPO 요건**: 데이터 손실 없음 → RPO ≈ 0. 실시간 복제를 통해 이를 충족하는 것은 액티브-액티브뿐입니다.
   - **RTO 요건**: 30분의 서비스 중단도 극도로 비용이 큼 → RTO ≈ 0. 액티브-액티브는 장애 발생 시 수 초 만에 다른 리전으로 트래픽을 전환합니다.
   - 충분한 예산이 여러 리전에서 전체 프로덕션 환경을 운영하는 높은 비용을 정당화합니다.

2. **핵심 AWS 서비스**:
   - **Route 53**: 레이턴시 기반 라우팅(latency-based routing) 또는 헬스 체크(health check) 장애 조치로 리전 간 트래픽 분산
   - **DynamoDB Global Tables**: 리전 간 멀티 액티브 데이터베이스 복제로 거의 제로에 가까운 RPO 달성
   - **Aurora Global Database**: 관계형 데이터가 필요한 경우, 1초 미만의 복제 지연(replication lag)
   - **AWS Global Accelerator**: 일관된 저지연 글로벌 라우팅
   - **S3 Cross-Region Replication**: 오브젝트 스토리지 자산 복제

</details>

### 연습 문제 4: NIST 특성의 실제 적용

각 시나리오에서 어떤 NIST 클라우드 컴퓨팅 필수 특성(essential characteristic)이 구현되고 있는지 찾아보세요:

1. 이커머스 사이트가 블랙 프라이데이(Black Friday) 기간에 서버 50대를 자동으로 추가하고 이후 다시 줄입니다.
2. 개발자가 누구에게도 연락하지 않고 AWS 콘솔에서 3분 만에 새 PostgreSQL 데이터베이스를 프로비저닝(provisioning)합니다.
3. AWS가 월별 스토리지 사용량을 정확한 GB-시간 단위로 측정하여 상세 청구 대시보드에 표시합니다.
4. 여러 고객의 워크로드(workload)가 동일한 물리 서버에서 실행되지만, 각 고객은 논리적으로 격리된 리소스를 사용합니다.

<details>
<summary>정답 보기</summary>

1. **신속한 탄력성(Rapid Elasticity)** — 시스템이 수요에 맞춰 리소스를 자동으로 확장하고 축소합니다.
2. **온디맨드 셀프 서비스(On-Demand Self-Service)** — 개발자가 공급자와의 상호작용 없이 리소스를 프로비저닝합니다.
3. **측정 가능한 서비스(Measured Service)** — 사용량이 투명하게 모니터링, 제어, 보고되어 사용량 기반 과금(pay-per-use)이 가능합니다.
4. **리소스 풀링(Resource Pooling)** — 물리적 리소스가 여러 테넌트(tenant)에게 공유되며(멀티 테넌시, multi-tenancy), 논리적 격리가 제공됩니다.

</details>

### 연습 문제 5: 비용 모델(Cost Model) 분석

한 회사가 세 가지 워크로드에 대해 온디맨드(On-Demand), 예약 인스턴스(Reserved Instances), 스팟 인스턴스(Spot Instances) 중 하나를 선택해야 합니다. 각 워크로드에 가장 적합한 가격 옵션을 추천하고 이유를 설명하세요:

1. 중단 없이 최소 2년간 24/7 운영해야 하는 프로덕션 웹 서버
2. 매일 밤 3시간 실행되며 중단 시 재시작 가능한 배치(batch) 데이터 처리 작업
3. 향후 2주 동안 불규칙하게 실행될 개발 환경의 새 마이크로서비스(microservice)

<details>

1. **예약 인스턴스(Reserved Instances, 1년 또는 3년 약정)** — 워크로드가 알려진 기간 동안 지속적으로 실행됩니다. 선불 약정으로 온디맨드 대비 최대 72%를 절약할 수 있습니다. 중단이 불가하므로 스팟은 적합하지 않습니다.

2. **스팟 인스턴스(Spot Instances)** — 중단을 허용하고 재시작이 가능한 배치 작업은 스팟의 이상적인 사용 사례입니다. 최대 90% 할인으로 특정 시간에 보장된 가용성이 필요하지 않은 워크로드의 비용을 크게 줄일 수 있습니다.

3. **온디맨드(On-Demand)** — 개발/테스트 워크로드는 사용 패턴이 예측 불가능하고 단기간입니다. 예약 인스턴스의 약정 혜택이 없으며, 테스트 도중에는 중단이 불가능할 수 있습니다. 온디맨드가 최대한의 유연성을 제공합니다.
