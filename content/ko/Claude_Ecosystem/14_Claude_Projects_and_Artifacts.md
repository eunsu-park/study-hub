Claude Projects와 Artifacts는 Claude 웹 및 데스크톱 인터페이스의 기능으로, Claude를 상태 없는 챗봇에서 영구적인 컨텍스트 인식 작업 공간으로 변환합니다. Projects는 자신의 문서와 코드에 Claude의 응답을 기반으로 하게 해주고, Artifacts는 Claude가 편집, 반복, 내보낼 수 있는 독립적인 콘텐츠 조각을 생성하게 해줍니다. 이 레슨에서는 두 기능을 심층적으로 다루며, 엔지니어링 워크플로우에서 Projects와 CLAUDE.md를 언제 사용할지도 설명합니다.

**난이도**: ⭐

**사전 요구 사항**:
- Claude에 대한 기본적인 친숙함 (웹 인터페이스 또는 데스크톱 앱)
- LLM이 컨텍스트를 사용하는 방법에 대한 이해 (도움이 되지만 필수는 아님)

## 학습 목표(Learning Objectives)

이 레슨을 완료하면 다음을 할 수 있습니다:

6. 다양한 사용 사례에 대해 Projects와 CLAUDE.md 중 선택할 수 있다
7. 프로젝트에서 지식을 구성하는 모범 사례를 적용할 수 있다

## 1. Claude Projects란 무엇인가?

Claude의 **Project**는 대화 간에 지속되는 정리된 작업 공간입니다. 다음을 포함하는 폴더라고 생각하면 됩니다:

- **프로젝트 지식(Project knowledge)**: Claude가 참조할 수 있는 문서, 코드 파일, 데이터
- **사용자 정의 지시사항(Custom instructions)**: Claude가 어떻게 동작해야 하는지에 대한 시스템 수준 가이드
- **대화(Conversations)**: 프로젝트의 컨텍스트를 공유하는 채팅 스레드
- **팀 접근(Team access)**: 공유 프로젝트를 위한 협업 설정

```
┌─────────────────────────────────────────────────────────────────┐
│                    Claude Project                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────────┐   ┌──────────────────────────────────┐        │
│  │   프로젝트   │   │        프로젝트 지식              │        │
│  │   설정       │   │  ┌────────────────────────────┐  │        │
│  │              │   │  │  design-spec.md             │  │        │
│  │  이름        │   │  │  api-schema.json            │  │        │
│  │  설명        │   │  │  style-guide.pdf            │  │        │
│  │  사용자 정의 │   │  │  database-schema.sql        │  │        │
│  │  지시사항    │   │  │  component-library.tsx      │  │        │
│  └──────────────┘   │  └────────────────────────────┘  │        │
│                      └──────────────────────────────────┘        │
│                                                                  │
│  ┌───────────────────────────────────────────────────────┐      │
│  │                      대화                             │      │
│  │  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐     │      │
│  │  │ 채팅 #1:    │ │ 채팅 #2:    │ │ 채팅 #3:    │     │      │
│  │  │ API 설계    │ │ 버그 수정   │ │ 코드 리뷰   │     │      │
│  │  └─────────────┘ └─────────────┘ └─────────────┘     │      │
│  └───────────────────────────────────────────────────────┘      │
│                                                                  │
│  모든 대화는 프로젝트 지식과 지시사항을 공유합니다               │
└─────────────────────────────────────────────────────────────────┘
```

**Projects를 사용하는 이유:**
- **영구적인 컨텍스트**: 문서를 한 번 업로드하고 많은 대화에서 참조
- **일관성(Consistency)**: 사용자 정의 지시사항으로 Claude가 매번 동일하게 동작하도록 보장
- **정리(Organization)**: 관련 대화를 함께 그룹화
- **협업(Collaboration)**: 팀 멤버와 프로젝트 공유 (Team 및 Enterprise 플랜)

## 2. 프로젝트 생성 및 관리

### 2.1 프로젝트 생성

claude.ai 또는 Claude Desktop에서:

1. 사이드바에서 **Projects** 아이콘을 클릭합니다 (또는 Desktop에서 `P` 누르기).
2. **Create Project**를 클릭합니다.
3. 이름과 선택적 설명을 입력합니다.
4. 프로젝트 지식 추가 (문서, 코드 파일).
5. 사용자 정의 지시사항 작성 (선택 사항).
6. 프로젝트 내에서 대화를 시작합니다.

### 2.2 프로젝트 구성 전략

범위와 목적에 따라 프로젝트를 구성합니다:

```
추천 프로젝트 구조:
├── 제품 프로젝트
│   ├── "Mobile App v3"      ← 제품/기능별 하나의 프로젝트
│   ├── "Admin Dashboard"
│   └── "Payment System"
│
├── 역할 기반 프로젝트
│   ├── "Technical Writer"   ← 작문 스타일 가이드가 있는 프로젝트
│   ├── "Code Reviewer"      ← 코딩 표준이 있는 프로젝트
│   └── "Data Analyst"       ← 스키마 문서 + SQL 패턴이 있는 프로젝트
│
├── 학습 프로젝트
│   ├── "Rust Learning"      ← Rust 책 발췌문이 있는 프로젝트
│   └── "ML Fundamentals"    ← 강의 노트가 있는 프로젝트
│
└── 참조 프로젝트
    ├── "Company Standards"  ← 스타일 가이드, 정책이 있는 프로젝트
    └── "API Reference"      ← API 문서가 있는 프로젝트
```

### 2.3 프로젝트 관리

- 자주 사용하는 프로젝트에 **별표**를 붙여 빠른 접근.
- 완료된 프로젝트를 **보관**하여 혼잡함을 줄입니다 (검색 가능한 상태로 유지됨).
- 더 이상 필요 없는 프로젝트를 **삭제**합니다 (영구적).
- 유사한 워크로드를 위한 템플릿을 만들려면 프로젝트를 **복제**합니다.

## 3. 지식 기반(Knowledge Grounding)

지식 기반은 Projects를 강력하게 만드는 핵심 기능입니다. 프로젝트에 문서를 업로드하면 Claude가 훈련 데이터만이 아니라 특정 정보에 직접 접근할 수 있습니다.

### 3.1 문서 추가

다음을 통해 지식을 추가할 수 있습니다:
- **파일 업로드**: 파일을 프로젝트 지식 영역에 드래그 앤 드롭
- **텍스트 붙여넣기**: 콘텐츠를 지식 항목으로 직접 붙여넣기
- **URL** (제한적): 일부 통합에서는 URL에서 가져오기 허용

### 3.2 지원되는 형식 및 제한

```
┌──────────────────────────────────────────────────────────────────┐
│                   지원되는 파일 형식                              │
├──────────────────┬───────────────────────────────────────────────┤
│ 형식             │ 참고사항                                       │
├──────────────────┼───────────────────────────────────────────────┤
│ PDF (.pdf)       │ 텍스트 추출; 스캔 PDF는 제한이 있을 수 있음   │
│ Text (.txt)      │ 일반 텍스트, 완전 지원                        │
│ Markdown (.md)   │ 렌더링 및 검색 가능                           │
│ 코드 파일        │ .py, .js, .ts, .java, .c, .cpp, .rs, 등      │
│ JSON (.json)     │ 구조화된 데이터로 파싱                         │
│ CSV (.csv)       │ 표 형식 데이터, 10k 행 이하가 최적            │
│ HTML (.html)     │ 텍스트 콘텐츠 추출                            │
│ DOCX (.docx)     │ Word 문서, 텍스트 추출                        │
│ XLSX (.xlsx)     │ 스프레드시트, 표 형식으로 변환                 │
│ 이미지           │ .png, .jpg, .gif, .webp (시각적 분석)        │
└──────────────────┴───────────────────────────────────────────────┘

제한 (2026년 초 기준):
- 개별 파일 크기:   최대 ~30 MB
- 총 프로젝트 지식: 최대 ~200k 토큰 (~500 페이지 텍스트)
- 파일 수:          실용적 제한 약 50-100 파일
- 파일 수 제한은 플랜(Pro, Team, Enterprise)에 따라 다름
```

### 3.3 Claude가 프로젝트 지식을 사용하는 방법

프로젝트 내에서 대화를 시작하면 Claude는 모든 프로젝트 지식을 컨텍스트 창의 일부로 받습니다. 이는 다음을 의미합니다:

1. **전체 컨텍스트 접근**: Claude가 업로드된 문서의 모든 부분을 참조 가능.
2. **문서 간 추론(Cross-document reasoning)**: Claude가 여러 파일에 걸쳐 정보를 연결 가능.
3. **기반이 있는 응답**: Claude가 훈련 데이터에만 의존하는 것이 아니라 문서를 인용하고 참조.
4. **훈련 데이터에 대한 우선순위**: 프로젝트 지식이 Claude의 훈련과 충돌할 때 프로젝트 지식이 우선.

```
┌──────────────────────────────────────────────────────────┐
│            프로젝트 지식 흐름                             │
│                                                          │
│  프로젝트 지식                                           │
│  ┌────────────────┐                                      │
│  │ design-spec.md │──┐                                   │
│  │ api-schema.json│──┤   ┌──────────────────┐           │
│  │ style-guide.md │──┼──▶│  시스템 컨텍스트  │           │
│  └────────────────┘  │   │  (모든 메시지와   │           │
│                      │   │   함께 전송됨)    │           │
│  사용자 정의 지시사항│   └────────┬─────────┘           │
│  ┌────────────────┐  │            │                      │
│  │ "당신은..."    │──┘            ▼                      │
│  └────────────────┘      ┌──────────────────┐           │
│                          │  사용자 메시지    │           │
│                          └────────┬─────────┘           │
│                                   │                      │
│                                   ▼                      │
│                          ┌──────────────────┐           │
│                          │  Claude의 응답   │           │
│                          │ (프로젝트 문서에  │           │
│                          │  기반한 응답)     │           │
│                          └──────────────────┘           │
└──────────────────────────────────────────────────────────┘
```

### 3.4 효과적인 지식 구성

프로젝트 지식의 가치를 극대화하는 팁:

```markdown
# 좋은 예: 명확한 섹션이 있는 구조화된 문서
## API 엔드포인트

### POST /api/users
새 사용자 계정을 생성합니다.

**요청 본문:**
| 필드     | 타입   | 필수 여부 | 설명                 |
|----------|--------|-----------|----------------------|
| name     | string | 예        | 사용자 전체 이름      |
| email    | string | 예        | 유효한 이메일 주소    |
| role     | string | 아니오    | 기본값: "viewer"     |

**응답 (201):**
```json
{
  "id": "usr_abc123",
  "name": "Alice Smith",
  "email": "alice@example.com",
  "role": "viewer",
  "createdAt": "2026-01-15T10:30:00Z"
}
```

**오류 응답:**
- 400: 유효성 검사 오류 (필수 필드 누락)
- 409: 이미 사용 중인 이메일
- 500: 내부 서버 오류
```

헤더, 테이블, 명확한 구분 없이 구조화되지 않은 텍스트를 쏟아내는 문서와 대조해보세요. Claude는 둘 다 처리할 수 있지만, 구조화된 문서가 훨씬 더 좋은 결과를 가져옵니다.

## 4. 사용자 정의 지시사항

사용자 정의 지시사항은 프로젝트 내 모든 대화에 대한 시스템 프롬프트 역할을 합니다. Claude의 동작, 어조, 초점, 출력 형식을 형성합니다.

### 4.1 사용자 정의 지시사항 설정

프로젝트 설정으로 이동하여 "Custom Instructions" 필드에 지시사항을 입력합니다. 이것은 프로젝트의 모든 새 대화에 적용됩니다.

### 4.2 지시사항 범주

**역할 기반 지시사항:**

```
당신은 Acme Corp의 시니어 백엔드 엔지니어입니다. Python(FastAPI)과 PostgreSQL을 전문으로 합니다. 코드를 검토할 때 다음에 집중합니다:
- 타입 안전성과 Pydantic 모델 사용
- SQL 인젝션 방지
- 구조화된 로깅과 적절한 오류 처리
- 성능 영향 (N+1 쿼리, 누락된 인덱스)

새 코드를 작성할 때 항상 포함합니다:
- 모든 함수 서명에 타입 힌트
- Args/Returns/Raises가 있는 독스트링
- 단위 테스트 제안
```

**어조 및 스타일 지시사항:**

```
명확하고 직접적인 스타일로 작성합니다. 사용자가 먼저 사용하지 않는 한 전문 용어를 피합니다.
기술적 개념을 설명할 때 일상 생활의 비유를 사용합니다.
사용자가 달리 지정하지 않는 한 항상 Python으로 코드 예제를 제공합니다.
응답을 간결하게 유지합니다 — 완전성보다 명확성을 목표로 합니다.
무언가에 대해 확실하지 않을 때는 명시적으로 말합니다.
```

**산업별 지시사항:**

```
당신은 의료 데이터 분석가입니다. 데이터 작업 시:
- 원시 PHI(보호된 건강 정보)를 표시하거나 출력하지 않습니다
- 예제에는 항상 비식별화된 샘플 데이터를 사용합니다
- 관련 있는 경우 HIPAA 준수 요구사항을 참조합니다
- 진단을 논의할 때 ICD-10 코드를 사용합니다
- 통계 분석에는 R, 데이터 엔지니어링에는 Python을 선호합니다
```

**출력 형식 지시사항:**

```
문서 검토를 요청할 때 다음 형식을 사용합니다:

## 요약
[문서에 대한 1-2문장 요약]

## 강점
- [글머리 기호 목록]

## 발견된 문제
| # | 심각도 | 위치 | 문제 | 제안 |
|---|--------|------|------|------|
| 1 | 높음   | p.3  | ...  | ...  |

## 권장 조치
1. [번호 매기기, 우선순위 목록]
```

### 4.3 지시사항 우선순위

여러 지시사항이 있을 때 (프로젝트 지시사항 + 대화 컨텍스트), Claude는 다음 우선순위를 따릅니다:

1. **프로젝트 사용자 정의 지시사항** (가장 높은 우선순위 — 시스템 수준)
2. **대화에서 사용자 메시지** (재정의 또는 구체화 가능)
3. **Claude의 기본 동작** (폴백)

이는 프로젝트 지시사항이 모든 대화에서 일관된 동작을 적용하는 데 신뢰할 수 있음을 의미합니다.

## 5. Artifacts

### 5.1 Artifacts란 무엇인가?

Artifacts는 Claude가 대화 중에 생성하는 독립적인 콘텐츠 조각입니다. 인라인 텍스트 응답과 달리 아티팩트는 별도 패널에 표시되며 다음이 가능합니다:

- 대화와 독립적으로 보기
- 편집 및 반복
- 다운로드 또는 복사
- 실시간 렌더링 (HTML, React, SVG의 경우)

아티팩트는 Claude의 "캔버스" — 구조화된 출력을 생성하기 위한 전용 공간이라고 생각하면 됩니다.

### 5.3 아티팩트 생성

아티팩트는 대화에서 자연스럽게 생성됩니다. Claude는 콘텐츠가 다음과 같을 때 아티팩트를 만들기로 결정합니다:
- **상당한 분량** (몇 줄 이상)
- **독립적** (자체적으로 의미가 있음)
- **재사용 가능성이 높음** (코드, 문서, 디자인)

아티팩트를 명시적으로 요청할 수도 있습니다:

```
사용자: 세 가지 티어에 대한 반응형 가격 테이블 HTML 아티팩트를 만들어 주세요:
        Basic ($9/월), Pro ($29/월), Enterprise (맞춤 견적).

Claude: [완전히 스타일링된 가격 테이블이 있는 HTML 아티팩트 생성]
```

### 5.4 편집 및 반복

아티팩트의 가장 강력한 기능 중 하나는 반복적 개선입니다:

```
대화 흐름:
─────────────────
사용자: "할 일 목록을 관리하는 Python 클래스 만들어 줘"
Claude: [아티팩트 생성: 추가/제거/완료 메서드가 있는 TodoList 클래스]

사용자: "우선순위 수준(낮음, 중간, 높음)과 정렬 기능 추가해 줘"
Claude: [우선순위 열거형과 정렬 메서드로 동일한 아티팩트 업데이트]

사용자: "JSON 직렬화와 파일 지속성 추가해 줘"
Claude: [to_json/from_json 및 save/load 메서드로 아티팩트 다시 업데이트]

사용자: "이 클래스에 대한 단위 테스트 작성해 줘"
Claude: [새 아티팩트 생성: pytest 테스트가 있는 test_todo_list.py]
```

각 반복은 이전 버전을 기반으로 합니다. Claude는 아티팩트의 상태를 추적하고 처음부터 다시 작성하는 것이 아니라 점진적으로 변경을 적용합니다.

### 5.5 프로토타이핑을 위한 아티팩트 활용

아티팩트는 특히 웹 인터페이스에서 빠른 프로토타이핑에 탁월합니다:

```
사용자: "애니메이션 차트로 실시간 CPU 및 메모리 사용량을 보여주는
        대시보드를 구축해 줘. HTML/CSS/JS만 사용해 줘 (프레임워크 없이)."

Claude: [다음이 포함된 HTML 아티팩트 생성:
  - 대시보드 패널을 위한 CSS Grid 레이아웃
  - 캔버스 기반 애니메이션 선 차트
  - 시뮬레이션된 실시간 데이터 업데이트
  - 반응형 디자인
  - 다크 모드 지원 색상 구성표]
```

아티팩트는 미리보기 패널에서 즉시 렌더링되어 Claude를 떠나지 않고도 프로토타입을 보고 상호작용할 수 있습니다.

**React 컴포넌트 프로토타이핑:**

```
사용자: "드래그 앤 드롭, 진행 표시줄, 파일 유형 유효성 검사가 있는
        파일 업로드 영역을 위한 React 컴포넌트를 만들어 줘."

Claude: [다음이 있는 React 아티팩트 생성:
  - 시각적 피드백이 있는 드래그 앤 드롭 영역
  - 파일 유형 유효성 검사 (이미지, PDF, 최대 10MB)
  - 업로드 진행 시뮬레이션
  - 제거 버튼이 있는 파일 목록
  - 잘못된 파일에 대한 오류 상태]
```

### 5.6 아티팩트 제한 사항

- **영구 상태 없음**: 아티팩트는 세션 간에 저장되지 않습니다 (복사/다운로드 필요).
- **제한된 의존성**: React 아티팩트는 제한된 라이브러리가 있는 샌드박스 환경에서 실행됩니다.
- **백엔드 없음**: HTML/React 아티팩트는 클라이언트 사이드에서만 실행됩니다 — 외부 서버에 대한 API 호출 없음.
- **크기 제한**: 매우 큰 아티팩트 (수천 줄)는 잘릴 수 있습니다.
- **파일 시스템 접근 없음**: 아티팩트는 컴퓨터의 파일을 읽거나 쓸 수 없습니다.

- 저장소에서 코드를 적극적으로 개발할 때
- Claude가 코드를 읽고, 쓰고, 실행해야 할 때

- 팀이 Claude 플랜이 아닌 저장소를 통해 컨텍스트를 공유할 때

**둘 다 함께 사용하는 경우:**
- 고수준 논의를 위해 아키텍처 문서를 Project에 업로드
- 일상적인 코딩을 위해 저장소에서 CLAUDE.md 유지
- 계획에는 Project를 사용하고, 구현에는 Claude Code로 전환

## 8. 모범 사례

### 8.1 지식 구성

1. **설명적인 파일 이름 사용**: `api-endpoints-v3.md`가 `doc1.md`보다 낫습니다.
2. **헤더로 문서 구조화**: Claude는 구조화된 문서를 더 효과적으로 탐색합니다.
3. **파일을 집중적으로 유지**: 파일 하나에 하나의 주제. 10페이지 API 명세가 100페이지 모든것-문서보다 낫습니다.
4. **예시 포함**: 구체적인 예시(샘플 요청, 코드 스니펫)가 있는 문서가 더 나은 Claude 응답을 만듭니다.
5. **정기적으로 업데이트**: 구식 문서를 제거합니다. 낡은 컨텍스트는 낡은 답변을 만듭니다.

### 8.2 사용자 정의 지시사항

1. **구체적으로**: "Python 코드 작성"은 약합니다. "타입 힌트가 있는 Python 3.12+ 코드 작성, 웹 엔드포인트에는 FastAPI, 데이터베이스 접근에는 SQLAlchemy 2.0 사용"은 강합니다.
2. **하지 말아야 할 것 정의**: "JavaScript 솔루션을 제안하지 마세요"는 불필요한 방향 전환을 방지합니다.
3. **출력 형식 기대치 설정**: Claude에게 응답이 어떻게 구조화되길 원하는지 정확히 알려줍니다.
4. **테스트 및 반복**: 대화를 시작하고 Claude가 어떻게 동작하는지 확인하고 지시사항을 개선합니다.
5. **지시사항을 1000단어 미만으로 유지**: 지나치게 긴 지시사항은 효과를 희석시킵니다.

### 8.3 Artifacts

1. **단순하게 시작하고 반복**: 먼저 기본 버전을 요청한 다음 기능을 점진적으로 추가합니다.
2. **요구사항에 대해 구체적으로**: "세 가지 티어가 있는 반응형 가격 테이블 만들어 줘"가 "가격 페이지 만들어"보다 낫습니다.
3. **중요한 아티팩트 저장**: 보관하고 싶은 아티팩트를 복사하거나 다운로드합니다 — 세션 간에 지속되지 않습니다.
4. **탐색에 아티팩트 활용**: 여러 접근 방식을 별도의 아티팩트로 프로토타이핑한 다음 최선의 것을 선택합니다.
5. **프로젝트 지식과 결합**: 디자인 시스템 문서를 업로드한 다음 Claude에게 그것을 따르는 아티팩트를 만들도록 요청합니다.

### 8.4 팀 협업

1. **프로젝트 소유자 지정**: 한 사람이 지식 기반과 지시사항을 유지해야 합니다.
2. **프로젝트 규약 문서화**: "이 프로젝트 사용 방법" 지식 항목을 추가합니다.
3. **공유된 대화 검토**: 팀 멤버가 프로젝트와 어떻게 상호작용하는지 살펴보고 지시사항을 개선합니다.
4. **관심사 분리**: 하나의 메가 프로젝트 대신 다른 워크스트림을 위한 다른 프로젝트를 만듭니다.

## 9. 연습 문제

### 연습 문제 1: 프로젝트 생성 (초급)

개인 취미나 관심사(요리, 사진, 피트니스 등)를 위한 Claude Project를 만듭니다:
1. 3-5개의 관련 문서 업로드 (레시피, 기술 가이드, 운동 계획).
2. Claude의 역할을 정의하는 사용자 정의 지시사항 작성 (개인 셰프, 코치 등).
3. 대화를 시작하고 Claude가 업로드된 문서를 참조하는지 확인합니다.
4. 여러 문서를 교차 참조해야 하는 무언가를 Claude에게 질문합니다.

### 연습 문제 2: 사용자 정의 지시사항 워크숍 (초급)

세 가지 다른 역할에 대한 사용자 정의 지시사항을 작성합니다:
1. Python Django 프로젝트를 위한 시니어 코드 리뷰어
2. API 문서를 만드는 기술 작가
3. 판매 데이터를 다루는 데이터 분석가

각각에 대해 역할 정의, 출력 형식, 피해야 할 것, 도메인별 요구사항을 포함합니다. 각각을 대화에서 테스트하고 개선합니다.

### 연습 문제 3: 아티팩트 프로토타이핑 (중급)

아티팩트를 사용하여 완전한 랜딩 페이지를 프로토타이핑합니다:
1. Claude에게 SaaS 제품 랜딩 페이지를 위한 HTML 아티팩트를 만들도록 요청합니다.
2. 반복적으로 추가합니다: 반응형 탐색, 히어로 섹션, 기능 그리드, 가격 테이블, 푸터.
3. Claude에게 가격 테이블만의 React 컴포넌트 버전을 만들도록 요청합니다.
4. 제품 로고를 위한 SVG 아티팩트를 요청합니다.
5. 모든 아티팩트를 내보내고 작동하는 프로젝트로 조립합니다.

### 연습 문제 4: 팀 프로젝트 설계 (중급)

REST API를 구축하는 5인 엔지니어링 팀을 위한 프로젝트 구조를 설계합니다:
1. 프로젝트 지식으로 업로드할 모든 문서를 나열합니다.
2. 사용자 정의 지시사항을 작성합니다.
3. 다른 팀 멤버(프론트엔드, 백엔드, QA, PM, 디자이너)가 프로젝트를 어떻게 사용할지 설명합니다.
4. 프로젝트 지식을 실제 코드베이스와 동기화 상태로 유지하는 방법을 설명합니다.

### 연습 문제 5: Projects + CLAUDE.md 통합 (고급)

풀스택 애플리케이션을 위해 Claude Projects와 CLAUDE.md를 함께 사용하는 워크플로우를 설계합니다:
1. Project에 들어갈 것(설계 문서, 아키텍처 결정, 회의록)을 정의합니다.
2. CLAUDE.md에 들어갈 것(코딩 표준, 빌드 명령, 테스트 패턴)을 정의합니다.
3. 팀 멤버가 어떤 도구를 언제 사용해야 하는지 설명하는 프로세스 문서를 작성합니다.
4. 서로 보완하는 샘플 CLAUDE.md와 매칭 Project 사용자 정의 지시사항을 만듭니다.
